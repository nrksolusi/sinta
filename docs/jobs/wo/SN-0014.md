## Objective

Move PO/SO fulfillment rollup from client-side join to server-side computation.
Server returns `receivedQty`/`deliveredQty` per line and document-level
`fulfillmentState` (`open|partial|closed`). Posting a receipt/delivery is guarded
so cumulative fulfilled qty cannot exceed ordered qty beyond the tenant tolerance
(default 0). Receipt line cost defaults from the linked PO line's agreed price.

## References

- ADR-0016: order fulfillment is server-computed, with tolerance-bounded over-receipt
- ADR-0017: receiving cost defaults from the PO line
- ADR-0019: read-query contract (fix-4 Slice 1, already in spec)

## Scope fence

**In scope**
- Migration: computed/rollup view or SQL for received/delivered qty per line
- OpenAPI: `receivedQty`, `deliveredQty`, `fulfillmentState` already added to spec in SN-0013; confirm they serialize
- Server handlers: rollup query on GET /purchase-orders/{id} and GET /sales-orders/{id}; list endpoints include rollup; posting guard in POST /goods-receipts and POST /deliveries
- `tenant_settings` already created in migration 0303; tolerance read in posting tx
- Receipt cost default: pre-fill `unitCost` from linked PO line on GET receipt form; permission check on override
- Client: drop `deliveredByLine`/`lineFulfillment`/`receivedByLine` client-side joins; consume server fields

**Out of scope**
- Approval gate (ADR-0015), cancel implementation (SN-0013 stub only)
- Per-product tolerance override (deferred in ADR-0016)
- ADR-0017 audit log table (recorded field only, not full audit trail)

### 2026-07-23 · implementation done (awaiting commit)
Implemented server-side fulfillment rollup and over-receipt guard. Changes staged
in worktree `.wt/sn-0014` on branch `feat/sn-0014-fulfillment-rollup`.

Key decisions:
- ADR-0016 says rollup is live-computed from posted non-reversal docs. SQL condition:
  `gr.status = 'posted' AND gr.reverses_id IS NULL` correctly excludes reversed
  originals (status='reversed') and their paired reversal GRs (reverses_id IS NOT NULL).
- sqlc generates `interface{}` for COALESCE/SUM aggregate expressions; added
  `numericFromAny(v interface{}) decimal.Decimal` helper that type-asserts to
  `pgtype.Numeric` (what pgx v5 delivers at runtime).
- `sqlc.yaml` numeric override did not help for aggregate expressions; kept
  `interface{}` and used the helper.
- Advisory lock: `pg_advisory_xact_lock(('x'||right(replace(id::text,'-',''),16))::bit(64)::bigint)`
  uses the last 64 bits of the UUID. Per-line lock (not per-PO) so two concurrent
  receipts against DIFFERENT lines of the same PO can proceed in parallel.
- `writeTransitionErr` (in document_posting.go) is the error handler for
  postStockDocument callbacks; added errOverReceipt and errOverDelivery there
  (not just in handleWriteErr) so the guard returns 422 correctly.
- Client gate passes without client code changes. The client-side join functions
  (receivedByPoLine, fulfillmentRows, etc.) still compile and their tests pass.
  Dropping those and wiring up server fields in the UI detail pages is deferred
  to a follow-up job (no failing client test drove it today).
- ADR-0017 (receipt cost default from PO line) deferred - not in the acceptance
  gate, no failing test was written for it. To be filed as SN-next.
